## Backend

Migrating your backend involves two key tasks: translating your Gen 1 resource definitions into Gen 2 TypeScript code, and transferring your stateful resources so the new environment shares the same data.

### Overall Approach

Migration is driven by a new `gen2-migration` subcommand in the Gen 1 CLI. It walks you through a series of steps that produce a new Gen 2 environment (called a branch) alongside your existing Gen 1 environment. The new branch gets fresh copies of your stateless resources (APIs, functions, roles) while your stateful resources (user pools, S3 buckets, DynamoDB tables) are transferred over so no data is lost. The following diagram describes the workflow:

<img src="/images/gen2-migration/workflow.png" alt="Migration workflow showing resource state at each step" width="900" />

After completing the migration, you will have two functionally equivalent Amplify applications that access the same data. You will no longer be able to update your Gen 1 environment — push updates to the Gen 2 code instead.

### Feature coverage

Before starting the migration, carefully review the [Backend Categories](/[platform]/start/migrate-to-gen2/feature-matrix/#backend-categories) section of the [Feature Parity](/[platform]/start/migrate-to-gen2/feature-matrix/) page. It will help you understand whether your environment can be fully migrated, what manual work may be required for partially supported or unsupported features, and whether there are any unknown areas that need further research on your part.

### Generating the Gen2 Application

The migration tool introspects your Gen 1 CloudFormation stacks and Amplify configuration files to produce the equivalent Gen 2 TypeScript backend definitions in the `amplify/` directory.

Features that are natively supported in Gen 2 are code-generated using the native Gen 2 APIs (e.g. `defineAuth`, `defineData`, `defineStorage`). Features that are not natively supported are code-generated as CDK constructs and [escape hatches](https://docs.aws.amazon.com/cdk/v2/guide/cfn-layer.html#develop-customize-escape).

Stateful resources (user pools, S3 buckets, etc.) are also code-generated, meaning new empty instances will be deployed in the Gen 2 environment. The refactor step will later replace these empty resources with your actual Gen 1 resources.

<Callout info>

DynamoDB tables that host your models are not cloned as part of the Gen 2 deployment and therefore do not participate in the refactoring step. Your Gen 2 application will have access to Gen 1 model data immediately after deployment.

</Callout>

The generated code can be deployed using either the Amplify Hosting service (via branch deployments) or the Gen 2 CLI (`npx ampx sandbox` / `npx ampx pipeline-deploy`).

### Preserving and Sharing Data

To preserve your existing data, the migration tool uses the [CloudFormation Stack Refactor](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/stack-refactoring.html) APIs to move your Gen 1 stateful resources (Cognito User Pool, Identity Pool, S3 bucket, etc.) into the Gen 2 CloudFormation stacks. The empty placeholder resources that were created during the generate phase are replaced with your actual Gen 1 resources, preserving all existing user accounts and data.

Stateless resources (APIs, functions, IAM roles) are not moved — the Gen 2 environment already has its own new instances of these. Your Gen 1 stateless resources remain in place so that any Gen 1 frontends continue to function until you switch them over. After refactoring, both your Gen 1 and Gen 2 environments access the same underlying data — changes made through either environment are immediately visible to the other.

<Callout info>

If you already have data replication mechanisms in place or prefer to set them up, you can use those to move data from your Gen 1 resources to the new Gen 2 resources instead of running the refactor step. Note that the migration tool focuses on code generation and stack refactoring — data replication is outside its scope and would need to be handled using your own tooling or AWS services.

</Callout>

## Frontend

Gen 1 frontends communicate with backend resources via the `amplifyconfiguration.json` file. All values in this file (AppSync endpoint URLs, User Pool IDs, etc.) remain valid throughout the entire migration process, so Gen 1 frontends continue to work without any change.

<img src="/images/gen2-migration/gen1-frontend-post-migration.png" alt="How existing Gen 1 frontends interact with backend resources post migration" width="320" />

Once you are satisfied the Gen 2 application works correctly, publish a new version of your frontend that connects to the Gen 2 stateless resources. In Gen 2, the configuration file has a different structure and is called `amplify_outputs.json`:

```ts
import amplifyconfig from '../amplify_outputs.json';
Amplify.configure(amplifyconfig);
```

Amplify client libraries detect the different structure and adjust automatically.

<img src="/images/gen2-migration/two-frontends-post-migration.png" alt="Both Gen 1 and Gen 2 frontends connecting to shared backend resources post migration" width="380" />

## Deploy & Host

> See the full [Deploy and Host](/[platform]/deploy-and-host/) documentation for comprehensive guidance on Gen 2 deployment options.

In Gen 1, deployment was typically done via `amplify push` (backend) and `amplify publish` (frontend) in a custom pipeline, or automatically through the Amplify Hosting service. In Gen 2, these commands no longer exist.

### Amplify Hosting

The simplest path is to connect your Git branch in the Amplify Console (_App Settings → Branch Settings → Add Branch_). The migration tool generates an `amplify.yml` buildspec file that enables Amplify Hosting to build and deploy your Gen 2 backend automatically on every push. See [branch deployments](https://docs.amplify.aws/react/deploy-and-host/fullstack-branching/branch-deployments/) for details.

<Callout info>

If you use a custom backend pipeline but still want Amplify Hosting for the frontend, update the `amplify.yml` build spec to run `npx ampx generate outputs` instead of `pipeline-deploy`, and use an incoming webhook to trigger the frontend build after your backend pipeline succeeds.

</Callout>

Since Amplify Hosting is able to deploy and publish both Gen 1 and Gen 2 environments within the same app, we recommend reusing your existing Amplify app rather than creating a new one. If you create a new app, any [custom domain](https://docs.aws.amazon.com/amplify/latest/userguide/custom-domains.html) configured on the original app cannot be reconfigured to point to the new migrated environment.

### Custom Pipelines

Unlike Gen 1, Gen 2 deployments always require an Amplify app ID and a branch name. If you use your own CI/CD system (AWS CodePipeline, Jenkins, CodeCatalyst, etc.) or a non-Git VCS like SVN, you can drive Gen 2 deployments directly using the `ampx` CLI. You will need to perform some one-time setup:

```bash
# (Optional) Create a new Amplify App — only needed if not reusing your existing app
aws amplify create-app --name my-app

# Register a branch name for your pipeline to deploy to
# (this is a logical name, not necessarily a real Git branch)
aws amplify create-branch --app-id <APP_ID> --branch-name gen2-main

# Disable auto-build so Amplify doesn't build on git push
aws amplify update-branch --app-id <APP_ID> --branch-name gen2-main --no-enable-auto-build
```

Then, in your pipeline build steps:

```bash
npm ci
export CI=1
npx ampx pipeline-deploy --branch gen2-main --app-id <AMPLIFY_APP_ID>
```

<Callout info>

The `--branch` parameter is a logical name, not a Git ref. You can use any string (e.g. `trunk`, `release-1.0`). The `amplify/` folder must be present in your working directory at build time.

</Callout>

See the [custom pipelines guide](/[platform]/deploy-and-host/fullstack-branching/custom-pipelines/) for full details.

### Frontend Hosting Without Amplify

If you host your frontend separately (S3 + CloudFront, on-prem, etc.), run `npx ampx generate outputs` in your pipeline to produce `amplify_outputs.json`, then bundle it with your frontend build artifacts.

## Prerequisites

### DataStore

If your API has conflict resolution enabled (DataStore), you must disable it before proceeding with the Gen 2 migration. See the [Migrate from DataStore](/gen1/[platform]/build-a-backend/more-features/datastore/migrate-from-datastore/) guide for instructions.

### GraphQL types protected by the `iam` auth provider

If you have models protected by IAM:

```graphql
type Todo @model @auth(rules: [{ allow: private, provider: iam }]) {
  id: ID!
  name: String!
  description: String
}
```

After refactor, the identity pool's `AuthRole` is updated to point to the Gen 2 role. Since this role is external to the Gen 1 AppSync API, it is denied access by default. Your Gen 2 environment will work correctly, but your Gen 1 environment will lose IAM access to the API.

To work around this, configure a custom admin role on the Gen 1 API that matches the Gen 2 `AuthRole` naming pattern. Add `amplify/api/<api-name>/custom-roles.json`:

```json
{
  "adminRoleNames": ["amplify-${appId}"]
}
```

Where `${appId}` is the Amplify application ID. Gen 2 auth role names are prefixed with this, so the pattern allows access from any Gen 2 environment (branch). Once added, redeploy by running `amplify push`.

### Amplify CLI Version

The migration tool relies on CloudFormation template structures and metadata produced by the most recent CLI major version. Deploy your Gen 1 environment with `v14` before migrating:

```bash
npm install -g @aws-amplify/cli@14
amplify push
```

### AWS Credentials

The migration tool requires the following API actions in addition to the standard Amplify CLI permissions:

- `cloudformation:CreateStackRefactor`
- `cloudformation:DescribeStackRefactor`
- `cloudformation:ExecuteStackRefactor`
- `cloudformation:GetStackPolicy`
- `cloudformation:SetStackPolicy`
- `cloudformation:DeleteChangeSet`
- `dynamodb:UpdateTable`
- `s3:GetBucketVersioning`
- `s3:GetEncryptionConfiguration`

<Callout info>

The managed `AdministratorAccess-Amplify` policy does not include these actions.

</Callout>

### CDK Readiness

Since Gen 2 uses CDK under the hood, your account and region must be [bootstrapped with CDK](https://docs.aws.amazon.com/cdk/v2/guide/bootstrapping-env.html) for the Gen 2 deployment to succeed.

## Modernization

The migration tool assumes a modern Gen 1 deployment. Outdated packages can cause peer dependency conflicts during `npm install` after `generate`. Upgrade the following before starting migration:

- **Node.js**: `aws-cdk-lib` requires Node.js >= 20. Ensure your local environment and CI/CD pipeline are running Node.js 20 or later.
- **aws-amplify**: Upgrade to `aws-amplify@^6.16.2` or later. If you are on v5, see the [v5 to v6 migration guide](https://docs.amplify.aws/react/build-a-backend/troubleshooting/migrate-from-javascript-v5-to-v6/).
- **@aws-amplify/ui-react**: If your app uses this package, upgrade to `^6`.
- **TypeScript**: Gen 2 generated code uses modern TypeScript features. Upgrade to `^5.0.0`.

## Step-By-Step Migration

<Callout info>

Before you begin, determine if your app can be migrated by reviewing the [feature coverage](#feature-coverage), [prerequisites](#prerequisites), and [modernization](#modernization) sections.

</Callout>

For clarity, the instructions below follow these conventions:

- Your Gen 1 environment is stored in the `main` branch of a Git repository.
- Your Gen 1 environment is called `main`.
- Deployment examples use Amplify Hosting (branch deployments), but this is not required. You can use any deployment method described in the [Deploy & Host](#deploy--host) section.

First, install an Amplify CLI version that includes the migration tool and pull a fresh copy of your Gen 1 environment:

```bash
npm install -g @aws-amplify/cli@^14.4.0
amplify pull --appId <app-id> --envName main
```

Verify your local project is in sync with the deployed environment:

```bash
amplify status  # Should report no changes
git diff        # Should show no changes
```

### Step 0: Clone

Before migrating your primary environment, perform the migration on a cloned environment first. Use `amplify env add` to create a clone of your Gen 1 environment, then check it out and deploy it:

```bash
# Environment names must be lowercase letters only and up to 10 characters
amplify env add
amplify env checkout <cloned-env-name>
amplify push
```

Run through the entire migration flow on this clone first. This lets you validate each step, catch configuration issues, and build confidence in the process. Once you are satisfied with the results, repeat the migration on your primary environment.

<Callout warning backgroundColor="var(--amplify-colors-red-10)">

**Environment Isolation**

When you clone an environment using `amplify env add`, the new environment deploys its own set of Amplify-managed resources. However, certain resources may still be shared between the original and the clone:

- Imported resources — If your app imports external resources (e.g. a Cognito User Pool or S3 bucket via `amplify import`), the cloned environment will reference the same underlying resource. Any operations performed during migration — such as refactoring or testing — could affect the original environment's data.
- Hardcoded references — Custom resources or CDK overrides that contain hardcoded ARNs, resource names, or account-specific identifiers will point to the same shared infrastructure across both environments.

<br />

**Review your cloned environment and ensure it is fully isolated before proceeding. Update any imported or hardcoded resource references to point to separate instances dedicated to the clone.**

</Callout>

### Step 1: Assess

Before starting the migration, evaluate whether your Gen 1 environment is ready:

```bash
npx amplify gen2-migration assess
```

This command is read-only with no side effects. It discovers all resources in your Gen 1 environment and produces a report showing migration support for each resource across the `generate` and `refactor` steps. If a resource or feature is not supported for `generate`, the tool skips it and you can manually add the configuration to the generated Gen 2 code. If not supported for `refactor`, you will need to manually move or migrate data from those resources.

<Callout info>

The `generate` and `refactor` steps also run this assessment as part of their validation and will fail if any entry is unsupported.

</Callout>

<Accordion title="Example assessment">

Running `assess` on an app with auth, a GraphQL API with an override, storage, and a Lambda function with custom IAM policies produces a report like this:


In this example, all resources are supported, but two features are flagged:

Both features show `—` for refactor because they only require code generation.

**Support indicators:** ✔ Supported (executed) · ✘ Unsupported (skipped) · — Not needed for this step (skipped)

</Accordion>

### Step 2: Lock

<Accordion title="Validations">

| Validation | Description |
|---|---|
| Environment Status | Verifies the root CloudFormation stack is in a stable state (`UPDATE_COMPLETE` or `CREATE_COMPLETE`). |
| Drift | Detects drift between your local project and the deployed CloudFormation stacks. |

</Accordion>

During the migration period your Gen 1 environment should not undergo any changes. Lock it by running:

```bash
npx amplify gen2-migration lock
```

Once validated, the following restrictive IAM policy is attached to the root stack:

```json
{ "Statement": [{ "Effect": "Deny", "Action": "Update:*", "Principal": "*", "Resource": "*" }] }
```

To unlock the environment, run:

```bash
npx amplify gen2-migration lock --rollback
```

<Callout warning>

Do not rollback the lock if the Gen 1 stack has already been refactored (and not rolled back). Pushing Gen 1 updates to a refactored stack can cause resource conflicts. It is also advisable to disable any automatic pipelines that deploy to your Gen 1 environment.

</Callout>


### Step 3: Generate

<Accordion title="Validations">

| Validation | Description |
|---|---|
| Lock Status | Verifies the Gen 1 environment is locked (deny-all stack policy is in place). |
| Working Directory | Verifies the git working directory has no uncommitted changes. |
| Assessment | Runs the resource and feature assessment for the `generate` step and fails if any entry is unsupported. |

</Accordion>

Generate your Gen 2 definition files:

```bash
git checkout -b gen2-main
npx amplify gen2-migration generate
```

This command introspects your deployed Gen 1 CloudFormation stacks and generates the equivalent Gen 2 TypeScript backend definition files, overriding your local `./amplify` directory. The generated code is a starting point — some features require manual edits before deploying (see below).

Once successful, clean up and reinstall dependencies to avoid stale resolution artifacts from the Gen 1 dependency tree:

```bash
rm -rf node_modules package-lock.json
npm install
npm install --package-lock-only
```

#### Post Generate | Frontend Config

If your frontend consumes `amplifyconfiguration.json`, update the import:

```diff
- import amplifyconfig from './amplifyconfiguration.json';
+ import amplifyconfig from '../amplify_outputs.json';
```

<Callout info>

The `amplify_outputs.json` file will not exist locally — it is generated at deploy time in the hosting service.

</Callout>

#### Post Generate | Reuse Model Tables

Edit `./amplify/data/resource.ts`:

```diff
- branchName: "main"
+ branchName: "gen2-main"
```

This instructs the hosting service to reuse (import) the DynamoDB tables instead of recreating them.

#### Post Generate | NodeJS Function ESM Compatibility

Port Lambda function code from CommonJS to ESM:

```diff
- exports.handler = async (event) => {
+ export async function handler(event) {
```

Gen 2 adds Lambda shims that conflict with CommonJS syntax.

#### Post Generate | Function Secrets

If your function uses secret values, recreate the secret in the Amplify console (_Hosting → Secrets → Manage Secrets → Add new_):

<img src="/images/gen2-migration/add-secret.png" alt="Adding a secret in the Amplify console" width="600" />

Then update the function definition:

```diff
- import { defineFunction } from "@aws-amplify/backend";
+ import { defineFunction, secret } from "@aws-amplify/backend";

- MY_SECRET: "/amplify/<hash>/main/AMPLIFY_<function-name>_MY_SECRET"
+ MY_SECRET: secret("/amplify/<hash>/main/AMPLIFY_<function-name>_MY_SECRET")
```

Then, in your function code, use `process.env.MY_SECRET` to obtain the secret value.

<Callout info>

See [Secrets](https://docs.amplify.aws/react/build-a-backend/functions/environment-variables-and-secrets/#secrets) for more information.

</Callout>

#### Post Generate | REST API

The migration tool generates the full REST API CDK construct automatically. Update your frontend code to point to the new Gen 2 API name:

```diff
- apiName: '<gen1-rest-api-name>',
+ apiName: '<gen2-rest-api-name>',
```

#### Post Generate | Models Without an `@auth` Directive

In Gen 1, unprotected `@model` types are considered public and assigned `@aws_api_key`. In Gen 2, they are considered private and assigned `@aws_iam`. To preserve the same behavior, explicitly add the `@auth` directive:

```graphql
type Todo @model @auth(rules: [{ allow: public }]) {
  id: ID!
  name: String!
  description: String
}
```

### Step 4: Deploy

Deploy the generated Gen 2 application via fullstack branch deployments. Push the code:

```bash
git add .
git commit -m "feat: migrate to gen2"
git push origin gen2-main
```

Then, log in to the AWS Amplify console and connect your new branch to the existing application (_App Settings → Branch Settings → Add Branch_). Wait for the deployment to complete.

<Accordion title="Testing with sandbox">

You can deploy using sandbox to test the Gen 2 application in full isolation from your Gen 1 environment:

```bash
npx ampx sandbox --once
```

By default, sandbox creates its own DynamoDB tables and does not share Gen 1 model data. To share them, set `branchName` to `"sandbox"` in `./amplify/data/resource.ts`.

</Accordion>


### Step 5: Refactor

<Accordion title="Validations">

| Validation | Description |
|---|---|
| Lock Status | Verifies the Gen 1 environment is locked (deny-all stack policy is in place). |
| Assessment | Runs the resource and feature assessment for the `refactor` step and fails if any entry is unsupported. |

</Accordion>

Refactoring is the process of moving your Gen 1 stateful resources (Cognito User Pool, S3 bucket, etc.) into the Gen 2 CloudFormation stacks using the [CloudFormation Refactor](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/stack-refactoring.html) APIs. After this step, both your Gen 1 and Gen 2 applications share the same underlying data and user accounts — your Gen 1 environment will no longer be updatable.

<Accordion title="Holding stacks">

During the forward refactor, the tool needs to move Gen 1 stateful resources into the Gen 2 CloudFormation stacks. However, the Gen 2 stacks already contain their own resources (created during the [Deploy](#step-4-deploy) step). CloudFormation cannot have two physical resources mapped to the same logical ID in a single stack, so the existing Gen 2 resources must be moved out of the way first.

To solve this, the tool creates temporary holding stacks — lightweight CloudFormation stacks that act as an intermediate destination for the Gen 2 resources. The forward refactor workflow proceeds in two phases per category:

1. Move Gen 2 resources → holding stack. The existing Gen 2 stateful resources (e.g. the User Pool, Identity Pool, or S3 bucket that were created during deploy) are transferred to a holding stack via the CloudFormation [StackRefactor](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/stack-refactoring.html) API. This is a metadata-only operation — the physical resources are not recreated or modified.
2. Move Gen 1 resources → Gen 2 stack. With the Gen 2 stack now clear of conflicting resources, the Gen 1 stateful resources are transferred in.

Holding stacks are named after the Gen 2 category stack they correspond to, with a `-holding` suffix (e.g. `amplify-d123-gen2main-branch-abc-auth-holding`).

Holding stacks are intentionally kept after a successful forward refactor. They enable rollback — if you run `refactor --rollback`, the tool moves Gen 1 resources back to the Gen 1 stacks and then restores the holding stack resources back into Gen 2, returning everything to its pre-refactor state.

</Accordion>

First, find the name of the Gen 2 root CloudFormation stack in the AWS CloudFormation console. Look for a stack with the pattern: `amplify-<appId>-gen2main-branch-<suffix>`.

<img src="/images/gen2-migration/find-stack.png" alt="Finding the Gen 2 root stack in the CloudFormation console" width="600" />

Then, reinstall the CLI package and run the refactor:

```bash
git checkout main
npx amplify gen2-migration refactor --to <gen2-root-stack-name>
```

<Callout info>

Since `generate` replaces the Gen 1 `amplify/` directory, the Amplify configuration file that specifies which AWS profile to use is lost. If you are using a non-default profile, set the `AWS_PROFILE` environment variable or run `amplify pull` to restore the configuration before proceeding.

</Callout>

If the refactor fails or produces undesired results, roll it back:

```bash
npx amplify gen2-migration refactor --to <gen2-root-stack-name> --rollback
```

After rolling back, you will need to unlock the Gen 1 environment and redeploy it to restore normal operation:

```bash
npx amplify gen2-migration lock --rollback
amplify push
```

### Step 6: Post-Refactor (CRITICAL)

<Callout warning backgroundColor="var(--amplify-colors-red-10)">

The following steps are critical. Skipping them can result in downtime.

</Callout>

The refactor step creates a drift between the deployed Gen 2 CloudFormation template and your local application code. This drift must be reconciled by a deployment. Additionally, the frontend configuration file (`amplify_outputs.json`) relies on stack output values that need to be updated to reflect the newly transferred resources — this also requires a deployment.

Switch to the Gen 2 branch for all post-refactor edits:

```bash
git checkout gen2-main
```

**S3 storage** — If your application contains an S3 bucket, edit `./amplify/backend.ts`:

```diff
- // s3Bucket.bucketName = '...';
+ s3Bucket.bucketName = '...';
```

**DynamoDB storage** — If your application contains a DynamoDB table in the storage category, edit `./amplify/backend.ts` to add the `tableName`:

```diff
- new Table(storageStack, "myTable", {
-   partitionKey: {...}
- });
+ new Table(storageStack, "myTable", {
+   tableName: 'my-table-main',
+   partitionKey: {...}
+ });
```

**Kinesis stream** — If your application contains a Kinesis stream, edit `./amplify/analytics/resource.ts`:

```diff
- // (analytics.node.findChild('KinesisStream') as CfnStream).name = "mystream-main"
+ (analytics.node.findChild('KinesisStream') as CfnStream).name = "mystream-main"
```

### Step 7: Deploy

Push the changes and wait for the deployment to finish:

```bash
git add .
git commit -m "fix: post refactor"
git push origin gen2-main
```

## Troubleshooting

### Node.js Dependencies Installation

Gen 1 Lambda functions may have dependencies declared in their `package.json` files. During `generate`, these are merged into the root `package.json` and can conflict with one another. If your Gen 2 deployment fails due to installation errors, inspect the root `package.json` and resolve conflicts manually.

### Circular CloudFormation Dependencies

Gen 1 apps where functions access multiple other resources (e.g. a query handler that also needs access to the data API, or an auth trigger that depends on storage) can produce circular dependencies between CloudFormation nested stacks when deployed as Gen 2. See [Troubleshoot circular dependency issues](https://docs.amplify.aws/vue/build-a-backend/troubleshooting/circular-dependency/) for resolution steps.
